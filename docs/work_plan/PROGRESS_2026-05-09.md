## Item 6 — Tighten Lighthouse budgets ⏸ DEFERRED

**Why deferred**:
The Lighthouse CI budgets in `.lighthouserc.json` are intentionally
**warn-only** today. We have not yet seen 5+ successful CI Lighthouse
runs to establish a realistic score baseline. Tightening to `error`
without baseline data = either overly strict (CI fails legitimate
deploys) or pointlessly lax (no actual gating).

**Re-trigger criteria**:
- ≥ 5 successful CI builds with Lighthouse scores in the logs
- Stable score range observed (e.g., performance always between X and Y)
- Then tighten one category at a time:
  - First: error if performance < (observed_min - 5)
  - Then: error if accessibility < (observed_min - 3)
  - etc.

**ETA**: ~2 weeks of normal commit activity should produce enough data.

## Item 7 — Dead code detection ⏸ DEFERRED

**Why deferred**:
Removing "unused" code that's actually loaded at runtime via reflection
(e.g., Firestore `fromJson` deserializers, lazy-imported screens via
`Navigator`, Riverpod-generated providers) breaks production.

A purely automated tool (`dart_code_metrics`, `unused_files`, etc.)
generates many false positives. Each candidate needs human review
to confirm it's actually unused.

**Re-trigger criteria**:
- Dedicated user review session — sit together, run the analyzer,
  go through each candidate, decide keep/delete
- Better as a "code health day" than as autonomous work

**Safe alternative for now**:
The Bundle Analysis Report (Item 4) already identifies likely dead-code
candidates (e.g., Stripe-related deps in `pubspec.yaml` that may be
stale). That's a manageable starting point for the user to review.
