# 📋 Work Plan — 2026-05-09

**Goal**: bring AnySkill closer to Uber/Airbnb/Wolt-grade quality through
test coverage, accessibility, observability, and documentation.

**Constraint**: only items I can complete autonomously (no payment provider,
no external account credentials, no Firebase Console actions).

**Status legend**: 🟡 in progress · ✅ done · ⏸ deferred · ❌ blocked

## Priority order (urgency × impact / effort)

### 🔥 HIGH priority — start immediately

#### 1. ✅ More Firestore Rules tests (uncovered collections)
- **Status**: ✅ done
- **Why first**: every uncovered collection = potential security regression. Tests are cheap (~5 min each), value is permanent.
- **Target collections** (TODO list before starting):
  - [x] `support_tickets/{id}/messages` channel filter (already partial coverage)
  - [x] `volunteer_requests` (admin-only updates)
  - [x] `category_requests` (custom-category approvals)
  - [x] `live_offers` (anytasks live offers — auth-only access)
  - [x] `dog_walks` doc-level rules
  - [x] `boarding_proofs/{jobId}/days/{dayKey}` daily rollup
  - [x] `motorcycle_tows` doc-level rules
  - [x] `flash_auctions/{id}/offers/{offerId}` offer write rule
  - [x] `community_requests` deeper coverage (claim/complete flow)
- **Output**: 1-2 new test files in `firestore-rules-tests/`, README updated

#### 2. ✅ More Semantics widgets (accessibility)
- **Status**: ✅ done
- **Why second**: regulatory (EU EAA 2025, IL חוק שוויון זכויות). Each missing
  Semantics = potential legal exposure once user count grows.
- **Target screens**:
  - [x] `home_tab.dart` — search bar, category cards (notification bell already done)
  - [x] `expert_profile_screen.dart` — service action buttons (Pay & Secure already done)
  - [x] `chat_screen.dart` — message input, send button
  - [x] `category_results_screen.dart` — expert cards, booking buttons
  - [x] `profile_screen.dart` — edit profile, logout, finance, settings
- **Output**: ~20 Semantics wrappers, code analyzer green

#### 3. ✅ CF tests with Firestore emulator (happy paths)
- **Status**: ✅ done
- **Why third**: currently we only test that bad inputs are rejected.
  Need to test that valid inputs actually work — otherwise an accidental
  bug (e.g., wrong field name in a job doc) won't be caught by tests.
- **Target CFs**: `processPaymentRelease`, `setUserRole`, `grantAdminCredit`,
  `processCancellation`, `notifyProviderOnApproval`
- **Output**: new test file `functions/__tests__/integration.test.js`
  using `firebase-functions-test` in **online** mode (real Firestore emulator)

### 📦 MEDIUM priority — do after high

#### 4. ✅ Bundle analysis report
- **Status**: ✅ done
- **Why**: establishes performance baseline. After deploy we know which
  packages are biggest and what to optimize.
- **Action**: run `flutter build web --analyze-size` → save the analysis
  output (which packages contribute most KB) to a markdown file.
- **Output**: `docs/work_plan/bundle_analysis_2026-05-09.md` with the
  size breakdown + top-10 heaviest deps.

#### 5. ✅ Code documentation (READMEs)
- **Status**: ✅ done
- **Why**: onboarding new developers, future-self, and audit trails.
  Several directories lack README.md:
  - [x] `tools/` — what's chromedriver and JRE21 doing here?
  - [x] `scripts/` — what each .js file does
  - [x] `firestore-rules-tests/` — already has README, just polish
  - [x] Project root — quick orientation for newcomers (already exists?)
- **Output**: 1-3 new README.md files

### 🌱 LOW priority — defer with criteria

#### 6. ⏸ Tighten Lighthouse budgets
- **Status**: ⏸ deferred (needs production baseline first)
- **Why deferred**: budgets are currently `warn` only. Can't tighten to
  `error` until we have 2-3 weeks of actual CI runs to see realistic scores.
- **Trigger**: revisit when there are ≥5 successful CI builds with
  Lighthouse scores in the logs. Tighten one category at a time.

#### 7. ⏸ Dead code detection
- **Status**: ⏸ deferred (high risk)
- **Why deferred**: removing "unused" code that's actually loaded at
  runtime via reflection (e.g., Firestore JSON deserializers) breaks prod.
- **Safe alternative**: only IDENTIFY candidates (write to a markdown file
  for manual review), do NOT delete. Will tackle in a future session
  with explicit user review.
