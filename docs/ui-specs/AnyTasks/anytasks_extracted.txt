AnyTasks
Implementation Prompt for Claude Code
Micro-Task Marketplace Module for AnySkill
Version 2.0 — April 2026
1. Project overview
AnyTasks is a micro-task marketplace module within the AnySkill app. It connects Clients (who need tasks done) with Service Providers (who earn money doing them). The module is designed with behavioral psychology embedded at every touchpoint to maximize engagement, retention, and transaction velocity.
Core principle
Clients PUBLISH tasks. Service Providers ACCEPT or COUNTER-OFFER. Payment is held in Escrow until the Client confirms completion. Every screen uses urgency, social proof, gamification, and progress psychology to drive action.
1.1 User roles
2. Task acceptance flow (critical logic)
This is the core interaction of AnyTasks. When a Service Provider sees a task, they have TWO options:
Option A: Direct accept (primary CTA)
Provider taps the green "Accept task" button
This means: "I agree to do this task at the Client’s listed price"
Client receives notification: "[Provider name] accepted your task!"
Client can view the provider’s profile (rating, badges, completion rate) and APPROVE or DECLINE
If Client approves → Escrow is charged → task begins
If Client declines → Provider is notified, task remains open
Option B: Counter-offer (secondary CTA, only if price doesn’t fit)
Provider taps "Suggest different price" (smaller/secondary button)
Opens a form: suggested price + short message explaining why
Client receives notification: "[Provider] suggests ₪[X] instead of ₪[Y] — [message]"
Client can Accept the counter-offer, Decline it, or Ignore it
UI priority: The "Accept task" button must be the DOMINANT CTA (large, green, full-width). The counter-offer option should be a secondary link/text below it, not a competing button.
3. Complete task lifecycle
4. Behavioral psychology layers
Every screen must embed at least 2-3 of these psychological triggers. They are NOT optional features — they are core product mechanics that drive growth.
4.1 Urgency + FOMO (Fear of Missing Out)
Task cards show: "Urgent — deadline in X hours" with countdown (coral badge)
"6 providers viewing this task right now" (live viewer count)
"4 offers already submitted" (competition counter for providers)
"Average task gets accepted within 45 minutes" (subtle urgency)
Push notification: "New task 800m from you! ₪150" (proximity + amount)
Counter-offer expiry: "Your offer expires in 2 hours" (time pressure on client)
4.2 Social proof
Star ratings prominently displayed (4.9 stars)
Completion rate percentage: "100% completion rate"
Task count: "47 tasks completed"
Verified badge (green checkmark) for identity-verified providers
Avatar stack showing faces of providers who responded
"[Name] completed a similar task and earned 5 stars"
4.3 Gamification
LEVELS: Beginner → Professional → Expert → Champion (based on completed tasks + rating)
STREAKS: "5 tasks in a row! Complete one more today for ₪25 bonus" (Duolingo-style)
BADGES: earned per specialty ("Delivery expert", "Photo pro", "Speed demon", "5-star")
PROGRESS BAR: "2 more tasks to reach Expert level" with visual bar at 91%
MONTHLY GOAL: "₪2,340 / ₪3,000 this month — 78%" with progress bar
LEADERBOARD: top providers in area by rating/completed tasks (weekly)
4.4 Goal gradient effect
Milestone timeline for active tasks: step 2 of 3 (brain resists abandoning mid-progress)
Profile completeness: "Your profile is 78% complete — add a photo to get 3x more offers"
Monthly earnings tracker with encouraging messages near the goal
4.5 Price anchoring + transparency
Client’s budget shown FIRST as anchor. Counter-offers appear relative to it ("Save ₪20")
Provider sees gross AND net: "₪120 → you receive ₪102 net" (full transparency)
Escrow badge on every task: green shield icon + "Payment secured" text
"0 publishing fee" shown at publish button to reduce friction
4.6 Reciprocity + commitment
First task published free (commitment trigger: once you’ve done it, you’re invested)
₪20 wallet credit for new providers (must complete a task to "cash out" → forces first action)
Referral: "Invite a friend → both get ₪15" with personal referral code
4.7 Hook model (habit loop)
TRIGGER: push notification about new matching task near provider
ACTION: open app, review task, tap Accept (made as easy as possible)
VARIABLE REWARD: will client choose me? (uncertainty = dopamine, like a slot machine)
INVESTMENT: rating, badges, level, reviews → creates sunk cost → hard to leave the platform

7. API endpoints
Tasks
POST /api/tasks — Client creates a new task
GET /api/tasks — Provider gets task feed (sorted by AI matching, filterable by category/distance/urgency)
GET /api/tasks/:id — Get task details
GET /api/tasks/my — Client: get my published tasks. Provider: get my active tasks
PATCH /api/tasks/:id/status — Update task status
Task responses
POST /api/tasks/:id/accept — Provider accepts at listed price (response_type: accept)
POST /api/tasks/:id/counter-offer — Provider suggests different price (response_type: counter_offer, body: { price, message })
GET /api/tasks/:id/responses — Client gets all responses for a task
POST /api/tasks/:id/choose/:response_id — Client chooses a provider (triggers Escrow)
Milestones
GET /api/tasks/:id/milestones — Get milestone timeline
PATCH /api/milestones/:id/complete — Provider marks step done (can include proof photo)
Payments
POST /api/escrow/create — Charge client, hold in escrow
POST /api/escrow/release/:task_id — Client confirms → release to provider
POST /api/disputes/:task_id — Open a dispute
Gamification
GET /api/provider/profile — Get level, badges, streaks, earnings
GET /api/provider/leaderboard — Weekly top providers
POST /api/referral/apply — Apply referral code
Notifications
Push notifications via Firebase Cloud Messaging (FCM). Real-time updates via WebSocket for chat and task status changes.
8. Platform fee logic
The platform charges a percentage fee from the final transaction amount. This fee is deducted from the Provider’s payout. The Client pays the full listed/agreed price.
Fee calculation: Provider receives = agreed_price - (agreed_price * platform_fee_percent). Example: Task at ₪120 with 15% fee → Provider gets ₪102, platform earns ₪18.
CRITICAL: The net amount must ALWAYS be shown to the provider before they accept. Display: "₪120 → you receive ₪102 net" on every task card and accept screen. This transparency builds trust and reduces disputes about money.
9. AI features
9.1 Task auto-complete
When client types a task title, AI auto-suggests: category, urgency level, budget range (based on similar tasks), relevant tags. Uses NLP to parse the description.
9.2 Provider matching
AI calculates a matching score for each task-provider pair based on: distance to task location, provider’s category expertise (badges), rating and completion rate, availability/streak status, historical response time. Score shown to Client as "AI recommends" badge.
9.3 Smart notifications
AI determines optimal notification timing per user (based on activity patterns). Personalizes notification text. Prioritizes high-match tasks. Avoids notification fatigue (max 5/day).
10. Anti-fraud and safety
Identity verification for providers (ID upload + selfie match)
Escrow prevents non-payment and ensures providers get paid
Photo proof with EXIF metadata validation (location + timestamp)
AI-powered fake review detection
Rate limiting on task responses (max 20 accepts per day per provider)
48-hour dispute resolution SLA with admin intervention
Provider suspension for completion rate below 70%
Client ban for repeated false disputes
11. Recommended tech stack
Frontend: React Native (existing AnySkill app). State management: React Context or Zustand. Backend: Node.js + Express or NestJS. Database: PostgreSQL with PostGIS for location queries. Real-time: WebSocket (Socket.io) for chat + status updates. Push notifications: Firebase Cloud Messaging. File storage: Cloudinary or S3 for proof photos. Payments/Escrow: Stripe Connect or local Israeli payment provider. AI/ML: OpenAI API for task parsing + matching algorithm.
12. Hebrew RTL support
All screens must support RTL layout. Text alignment: right. Navigation arrows: reversed. Category chips scroll from right. Form labels aligned right. Numbers remain LTR within RTL context.

6. Accept vs Counter-offer UI specification
This is the most critical interaction in the app. The UI must make ACCEPT the obvious, fast, frictionless default action. Counter-offer is available but intentionally requires one extra tap.
6.1 Provider task card (in feed)
Full-width GREEN button: "Accept task" — 48px height, 8px radius, white text 14px medium, green (#2D6A4F) background
Below the button (8px gap): centered text link in gray: "Suggest different price" — 13px, no underline, tappable area extends to full width for easy tap
Bookmark icon positioned at the top-right of the card
6.2 Provider task detail (full screen)
Bottom sticky section (white bg, top border, padding 16px):
GREEN button: "Accept task at ₪120" — full-width, 48px, same green styling
Below button: "You receive: ₪102 net" in small green text (11px)
Below that: "Price doesn’t work? Suggest a different amount" text link (gray, 12px)
Tapping the link expands an inline form (animated slide-down): Your price (number input, centered, 16px bold), Short message (textarea, 3 lines, "Why is this price fair for you?"), "Send offer" button (outline style, not filled, to keep visual hierarchy below Accept), Cancel link
6.3 Client comparison card
When client views responses, they see TWO types of cards side by side:
ACCEPT cards: green left border, checkmark icon, text "Accepted at ₪120" — these are instant, client just needs to approve the provider.
COUNTER-OFFER cards: amber left border, price-tag icon, text "Suggests ₪100 (save ₪20)" + provider’s message below — client can accept the counter-offer or skip.
Both card types show the same provider info (avatar, rating, badges, distance, completion %).
